# Incident: Booking API 500 — 2026-04-25 ~15:30 UTC+5

## What happened

`POST /api/v1/inquiries` returned HTTP 500 for ~1 request window.
The monitoring script caught it and sent a Telegram alert.

## Root cause

`CACHE_DRIVER=file` was in use. Laravel's `ThrottleRequests` middleware
writes rate-limit lock files into `storage/framework/cache/data/{hash}/`.
Those subdirectories were missing (likely wiped by a prior manual
`cache:clear`). `fopen()` on a non-existent path threw `ErrorException`
before the request reached any controller.

Stack trace: `ThrottleRequests → FileStore::add → LockableFile::__construct → fopen`.

## Fix applied

1. `CACHE_DRIVER=redis` — rate limiter and all cache writes now go to Redis.
   No filesystem directories required.
2. `SESSION_DRIVER=redis` — removed the same failure mode for session writes.
3. Cleared stale `REDIS_PASSWORD` from `.env` (Redis runs with no auth on
   `127.0.0.1`; password field was causing `ERR AUTH` connection failures).

## Detection improvement

`check-booking-api.sh` (cron every 30 min) now checks Redis first via
`redis-cli ping`. If Redis is down, a Telegram alert fires immediately —
before Laravel starts throwing 500s. Separate lock files prevent alert spam
for Redis vs. API failures independently.

## Prevention

- Runtime state (cache, sessions, rate limiting) no longer depends on filesystem.
- Redis is bound to `127.0.0.1` only — no external exposure.
- `REDIS_PASSWORD` should be set if Redis ever leaves localhost boundary (deferred).
